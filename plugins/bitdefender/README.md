# HiveArmor Bitdefender GravityZone Plugin

**Plugin name:** `com.hivearmor.bitdefender`
**Module:** `github.com/hivearmor/plugins/bitdefender`
**Language:** Go 1.25.5

## Overview

This plugin connects HiveArmor to Bitdefender GravityZone Cloud. It runs an HTTPS webhook server that receives security events from the GravityZone Event Push Service, validates them against configured company IDs, and forwards them to the HiveArmor Event Processor for correlation and indexing.

Events are ingested in CEF format and stored in OpenSearch under the data type `antivirus-bitdefender-gz`.

## How It Works

```
GravityZone Cloud
    │
    │  HTTPS POST (CEF events)
    ▼
Plugin webhook server  :8000/api
    │  Basic auth validation
    │  Company ID matching
    │
    │  gRPC log channel
    ▼
HiveArmor Event Processor
    │
    ▼
OpenSearch  _v3_hive_antivirus-bitdefender-gz-YYYY.MM.DD
```

### Startup sequence

1. The plugin connects to the HiveArmor modules-config service via a gRPC bidirectional stream and receives its configuration.
2. For each configured group (tenant / company set), it automatically calls the GravityZone API (`setPushEventSettings`) to register the webhook URL and subscribe to event types.
3. GravityZone begins pushing events to the plugin's HTTPS endpoint.
4. Incoming events are matched to their company ID, enqueued in an internal log channel, and drained by a pool of forwarding goroutines (2 × CPU count) that write to the Event Processor via the plugin SDK.

### Authentication

Webhook requests from GravityZone carry an `Authorization` header in the form:

```
Basic base64(<connectionKey>:)
```

The plugin derives the expected value from the configured `connectionKey` and rejects any request that does not match with HTTP 401.

### TLS

The webhook server requires TLS 1.2 or higher. It loads `ha.crt` and `ha.key` from the path configured in the HiveArmor plugin runtime (`certsFolder`). These certificates are generated by the HiveArmor installer during first-run setup. GravityZone's `requireValidSslCertificate` flag is set to `false` in the push-event registration because HiveArmor uses a self-signed certificate by default; replace the certificate with a CA-signed one in production environments if required by your security policy.

## Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api` | Receive CEF event batch from GravityZone |
| GET | `/status` | Health check — returns `200 OK` when the server is up |

Default port: **8000**

## Configuration

Configuration is delivered to the plugin at runtime via the HiveArmor configuration service. The following fields are required per group (one group per GravityZone account / MSSP tenant):

| Field | Key in config | Description |
|-------|---------------|-------------|
| Connection key | `connectionKey` | GravityZone API key used to authenticate webhook calls. Also used to register the push-event settings via the GravityZone JSON-RPC API. |
| GravityZone API URL | `accessUrl` | Base URL of the GravityZone API endpoint (e.g. `https://cloud.gravityzone.bitdefender.com`). |
| HiveArmor public IP | `utmPublicIP` | Public IP or hostname of the HiveArmor server that GravityZone will send webhooks to. The plugin registers `https://<utmPublicIP>:8000/api` with GravityZone automatically. |
| Company IDs | `companyIds` | Comma-separated list of GravityZone company IDs whose events this group should accept. Events not matching any listed company ID are silently discarded. |

Configuration is applied by the HiveArmor platform — you do not edit files on disk. To add or change a Bitdefender integration, update it through the HiveArmor UI under **Settings > Integrations > Bitdefender GravityZone**.

After a configuration change, the plugin waits 15 seconds, fetches the active push settings from GravityZone (`getPushEventSettings`) to confirm they were applied, then sends a test event (`sendTestPushEvent`) to verify end-to-end connectivity.

## Multi-tenant (MSSP) Support

A single plugin instance supports multiple GravityZone accounts via module groups. Each group has its own `connectionKey`, `accessUrl`, `utmPublicIP`, and `companyIds`. Incoming events are routed to the correct group by matching the `BitdefenderGZCompanyId` field in the CEF message against each group's configured company ID list.

## Building

```bash
cd plugins/bitdefender
go build -o com.hivearmor.bitdefender.plugin .
```

The output binary must be named `com.hivearmor.bitdefender.plugin` — this is the exact name the HiveArmor Event Processor uses to load the plugin.

## Troubleshooting

**Plugin does not start / exits immediately**
The plugin only runs when its operating mode is `manager`. If the mode is not set to `manager` in the HiveArmor runtime configuration, the process exits cleanly with no output. This is expected for non-manager nodes.

**No events arriving in HiveArmor**
1. Confirm the GravityZone push settings were registered: check plugin logs for `Received configuration update` and the absence of `error sending configuration`.
2. Verify that port 8000 is reachable from the GravityZone cloud IP ranges on the `utmPublicIP` host.
3. Confirm the `connectionKey` matches the API key used when the integration was configured in HiveArmor.
4. Check GravityZone push event delivery logs in the GravityZone console under **Control Center > Notifications > Event Push Service**.

**401 Unauthorized errors in plugin logs**
The `connectionKey` in HiveArmor does not match the value GravityZone is using to sign its webhook requests. Update the `connectionKey` in HiveArmor and allow up to one configuration sync cycle (approximately 15 seconds) for GravityZone to re-register.

**Certificate errors**
Ensure `ha.crt` and `ha.key` exist in the configured `certsFolder` path. These files are created by the HiveArmor installer. If they are missing, re-run the HiveArmor TLS certificate generation step or replace them with your own PEM-format certificate and key pair.
